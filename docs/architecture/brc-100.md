# BRC-100 Wallet Interface

BRC-100 defines a vendor-neutral boundary that separates application business logic from wallet and key management. Applications implement BRC-100 once and work with any compliant wallet (BSV Browser, BSV Desktop, `@bsv/wallet-toolbox`, or a custom implementation).

The authoritative machine-readable spec is at `specs/sdk/brc-100-wallet.json` (JSON Schema). This page is the human-readable counterpart.

## Method Surface

### Actions

**`createAction(params)`** — Build a transaction.

- `params.description` — Human-readable description shown in wallet UX
- `params.inputs` — UTXOs to spend (optional; wallet selects automatically if omitted)
- `params.outputs` — Outputs to create (locking scripts + satoshi amounts)
- `params.labels` — Transaction-level labels (see Labels vs. Tags below)
- `params.options.noSend` — If `true`, prepare the transaction but do not broadcast (see Batching)
- `params.options.sendWith` — List of `txid` references from previous `noSend` actions to broadcast together

Returns a `signableTransaction` object with a `.reference` handle and the unsigned/partially-signed transaction bytes.

**`signAction({ reference, spends })`** — Complete signing. Pass the `.reference` from `createAction` and the input unlock scripts.

**`abortAction({ reference })`** — Cancel an in-progress action. Releases reserved UTXOs.

**`listActions(params)`** — Query transactions by label. Returns action history with BEEF for each.

**`internalizeAction({ tx, outputs, description, labels })`** — Internalize an incoming BEEF transaction into the wallet (credit received funds, track baskets).

### Outputs

**`listOutputs(params)`** — Query wallet outputs (UTXOs). Key filters:
- `params.basket` — Basket name to query
- `params.tags` — Output-level tags to filter by (see Labels vs. Tags)
- `params.include` — Whether to include BEEF, locking scripts, etc.

**`relinquishOutput({ basket, output })`** — Release an output from a basket without spending it.

### Keys and Cryptography

**`getPublicKey(params)`** — Derive a public key using BRC-42. Specify `protocolID`, `keyID`, and `counterparty`.

**`encrypt({ plaintext, protocolID, keyID, counterparty })`** — Encrypt data. Returns ciphertext bytes.

**`decrypt({ ciphertext, protocolID, keyID, counterparty })`** — Decrypt data. Returns plaintext bytes.

**`createHmac({ data, protocolID, keyID, counterparty })`** — Create an HMAC over `data`.

**`verifyHmac({ data, hmac, protocolID, keyID, counterparty })`** — Verify an HMAC.

**`createSignature({ data, protocolID, keyID, counterparty })`** — Sign arbitrary data. Returns DER-encoded signature.

**`verifySignature({ data, signature, protocolID, keyID, counterparty, forSelf })`** — Verify a signature.

### Identity Certificates

**`acquireCertificate(params)`** — Request and store an identity certificate from an issuer.

**`listCertificates(params)`** — List held certificates, optionally filtered by certifier or type.

**`proveCertificate({ certificate, fieldsToReveal, verifier })`** — Generate a selective disclosure proof for specific certificate fields.

**`relinquishCertificate({ type, serialNumber, certifier })`** — Remove a certificate from the wallet.

**`discoverByIdentityKey({ identityKey })`** — Look up certificates for an identity key.

**`discoverByAttributes({ attributes })`** — Look up identities by attribute values.

### Wallet State

**`isAuthenticated()`** — Check if the wallet has an active session.

**`waitForAuthentication()`** — Block until authenticated.

**`getHeight()`** — Current chain tip block height.

**`getHeaderForHeight({ height })`** — Block header for a given height.

**`getNetwork()`** — `'mainnet'` or `'testnet'`.

**`getVersion()`** — Wallet implementation version string.

## Labels vs. Tags

These are frequently confused. They apply at different granularities:

| Concept | Scope | Purpose | Query method |
|---------|-------|---------|--------------|
| **Labels** | Transaction | Categorize whole Actions | `listActions({ label })` |
| **Tags** | Output (UTXO) | Track specific assets within a basket | `listOutputs({ tags })` |

Example: label a transaction `"subscription"` and tag individual outputs as `"monthly"` within a `"subscriptions"` basket.

## Batching Workflows

### `noSend` — Prepare without broadcasting

```typescript
const result = await wallet.createAction({
  description: 'Prepare token mint',
  outputs: [...],
  options: { noSend: true }
})
// result.signableTransaction.reference is a handle to the prepared tx
const preparedRef = result.signableTransaction.txid
```

The transaction is signed and ready but not broadcast. Useful for constructing coordinated multi-transaction sequences.

### `sendWith` — Couple and broadcast atomically

```typescript
await wallet.createAction({
  description: 'Final action, broadcast all',
  outputs: [...],
  options: {
    sendWith: [preparedRef1, preparedRef2]
  }
})
// All three transactions broadcast together
```

`sendWith` couples the current action with previously `noSend` actions, broadcasting them as an atomic group.

## Privileged Mode

All wallet keys are derived via BRC-42 (BKDS). The BRC-100 interface supports a secondary **Privileged Mode** keyring for operations requiring elevated trust:

- Identity key signing
- Certificate issuance
- High-value transaction authorization

Privileged operations require explicit wallet UX confirmation. The privileged keyring is isolated from application-level keys — privileged key material never surfaces to application code.

## BRC-100 and WalletClient

`@bsv/sdk` exports a `WalletClient` substrate that routes BRC-100 calls to the appropriate transport:

- **Browser** — Routes to `window.postMessage` or `localhost` socket (BSV Browser wallet)
- **Direct** — Routes to an in-process `Wallet` instance (e.g., from `@bsv/wallet-toolbox`)

`@bsv/simple/browser` wraps `WalletClient` with a higher-level API. `@bsv/simple/server` wraps `@bsv/wallet-toolbox`'s `Wallet` directly.
