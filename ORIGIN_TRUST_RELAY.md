# Proposal: Origin-Trust Relay Resolution

**Branch:** `proposal/origin-trust-relay`  
**Status:** Draft — open for discussion

---

## Problem

The library currently enforces an M1 security check: `wss://` relay hostnames must match the origin hostname. This was designed to prevent a malicious QR from redirecting a mobile wallet to an attacker-controlled relay.

The consequence is that the relay **must be hosted on the same domain as the web app**. This rules out:

- Serverless deployments (Vercel, Netlify) where persistent WebSocket servers cannot run
- Third-party managed relay services (Ably, Pusher, Soketi)
- Any architecture where the relay and the web app are on separate infrastructure

---

## Why a Signed QR Doesn't Solve It

One considered alternative was embedding a cryptographic signature in the QR URI, generated by the backend wallet (`wallet.createSignature`). The mobile would verify the signature before connecting.

**This does not work.** The signature only proves that *some wallet* generated the QR — it says nothing about *whose* wallet. Any attacker with their own wallet can produce a perfectly valid signed QR pointing to their own relay and their own backend key. The mobile would accept it, connect to the attacker's relay, and the attacker would hold the private key matching the `backendIdentityKey` in the QR.

Signatures provide integrity ("this data hasn't been tampered with") but not origin identity ("this came from myapp.com"). Without binding the key to a domain — which requires PKI — signatures alone cannot replace domain-level trust.

---

## Proposed Solution: Fetch Relay from Origin

This is the same pattern OAuth uses for its authorization endpoint: trust is established by making a direct HTTPS call to a URL the user can see and verify, rather than trusting data embedded in a QR code.

### New flow

```
1. Mobile scans QR
   QR contains: topic, origin, backendIdentityKey, protocolID, keyID, expiry
   QR no longer contains: relay URL

2. App shows params.origin to the user
   → "You are about to connect to: myapp.com"
   → User approves

3. Mobile calls session.resolveRelay()
   GET https://[origin]/api/session/:topic
   ← { relay: "wss://...", status: "pending" }
   Trust anchor: origin's TLS certificate (same PKI as the web)

4. Mobile calls session.connect()
   → WebSocket connects to the resolved relay URL
   → Pairing handshake proceeds as before (unchanged)
```

### Why this maintains security

- The relay URL is served by the origin over HTTPS — an attacker cannot forge this without controlling the origin domain's TLS certificate
- The user sees `params.origin` (the domain they recognise from the browser) as the trust anchor, not the relay infrastructure URL
- ECDH encryption still protects all message content — the relay (whether self-hosted or third-party) only ever sees ciphertext
- This is strictly equivalent to M1 in terms of the trust anchor, and more flexible in relay deployment

### Trust anchor comparison

| Approach | Trust anchor | Relay constraint |
|----------|-------------|-----------------|
| M1 (current) | relay hostname === origin hostname | Relay must be on the same domain |
| Origin fetch (proposed) | HTTPS TLS cert of origin | Relay can be anything the origin server returns |

## What Changed

### Breaking changes

This is a **breaking change** for any existing mobile app using `WalletPairingSession`.

| Component | Change |
|-----------|--------|
| `PairingParams` | `relay` field removed |
| `buildPairingUri` | `relayURL` param removed — relay is no longer in the QR |
| `parsePairingUri` | `relay` no longer required or validated; M1 check removed |
| `WalletPairingSession.connect()` | Requires `resolveRelay()` to be called first; throws otherwise |
| `WalletPairingSession.reconnect()` | Same — requires prior `resolveRelay()` call |
| `WalletRelayService.getSession()` | Now returns `relay` in its response |

### New API

**`WalletPairingSession.resolveRelay(): Promise<string>`**

Fetches the relay WebSocket URL from the origin server. Must be called before `connect()`. Returns the relay URL so the app can display it to the user.

```ts
const { params } = parsePairingUri(qrString)

// 1. Show origin to the user and wait for approval
showApprovalUI(params.origin)
await userApproved()

// 2. Fetch relay URL from origin (HTTPS — TLS is the trust anchor)
const relay = await session.resolveRelay()

// 3. Optionally show relay to the user before connecting
// 4. Connect
await session.connect()
```

## Mobile App Changes Required

Any app using `WalletPairingSession` needs to update its pairing flow:

**Before:**
```ts
const { params } = parsePairingUri(qrString)
const session = new WalletPairingSession(wallet, params, options)
session.onRequest(handler)
await session.connect()
```

**After:**
```ts
const { params } = parsePairingUri(qrString)
// Show params.origin to the user for approval
const session = new WalletPairingSession(wallet, params, options)
session.onRequest(handler)
await session.resolveRelay()  // fetches relay from origin over HTTPS
await session.connect()
```

## Discussion Points

1. **User approval UX** — The app must show `params.origin` to the user before calling `resolveRelay()`. This is important: if the app skips this step, the trust guarantee is degraded. Should the library enforce this somehow (e.g., `resolveRelay()` requires an explicit `{ userApproved: true }` flag), or is this left to the implementing app?

2. **Reconnect after app restart** — `resolveRelay()` is a lightweight HTTP call. For reconnects, the app should call it again before `reconnect(lastSeq)`. Alternatively, the app can persist the relay URL alongside `lastSeq` and set it directly if needed (though the API for this isn't currently exposed).

3. **Origin fetch failure** — If the origin server is unreachable at scan time, `resolveRelay()` will throw. The relay and origin are typically the same deployment, so in practice if the origin is up (QR was displayed) the fetch will succeed. Edge case: QR scanned offline. Worth documenting clearly.

4. **Backward compatibility** — Old QR codes that still include a `relay` param will have it silently ignored by the new `parsePairingUri`. The mobile will fetch the relay from origin regardless. This is safe since origin is the authoritative source.

5. **`relayUrl` in `WalletRelayServiceOptions`** — Still required for the service to know where its WebSocket server is. It is now returned via `GET /api/session/:id` rather than embedded in the QR.
