Upgrade Guide · v2.x.x → v3.0.0

Same wallet API.
Storage that finally tells the truth.

v3 collapses the legacy transactions + proven_tx_reqs + proven_txs triad into a clean two-table model, adds a 14-state FSM with a full audit log, enforces reorg-safe spendability, and ships horizontal Monitor scaling — all behind the exact same BRC-100 wallet interface your app already calls today.

Why upgrade

v2 ships. v3 scales.

The toolbox you run today works — until it doesn't. Reorgs that retroactively flip confirmed UTXOs to invalid. Coinbase outputs that go spendable before 100 blocks. A single Monitor instance that can't be load-balanced without state corruption. v3 fixes all of it, on the same hot-query plan.

01 — Schema: from 3 tables to 2 Single row per txid; per-user data moves to actions. ›

In v2, every transaction was scattered across three places: a per-user transactions row, a broadcast/processing row in proven_tx_reqs, and (eventually) a proof row in proven_txs. Status semantics were split across transactions.status and proven_tx_reqs.status, and reconciliation logic had to live in application code.

v3 promotes transactions to the single chain-canonical table — one row per txid — and introduces actions for everything that is truly per-user (description, label associations, hidden flag, notification subscribers, satoshis delta).

v2 — three tables

  • transactions · userId, status, satoshis, description, reference, isOutgoing, version, lockTime, inputBEEF, ...
  • proven_tx_reqs · txid, status, attempts, wasBroadcast, rawTx, inputBEEF, history (JSON), ...
  • proven_txs · txid, height, merklePath, merkleRoot, blockHash, rawTx
  • tx_labels_map · transactionId → transactions.transactionId

v3 — two canonical tables

  • transactions · txid, processing, processingChangedAt, nextActionAt, attempts, rebroadcastCycles, wasBroadcast, idempotencyKey, batch, rawTx, inputBeef, height, merklePath, merkleRoot, blockHash, isCoinbase, lastProvider, frozenReason, rowVersion
  • actions · userId, transactionId, reference, description, isOutgoing, satoshisDelta, userNosend, hidden, userAborted, notifyJson
  • tx_labels_map · transactionId → actions.actionId
  • tx_audit · new — every transition + reason
  • monitor_lease · new — multi-instance Monitor
  • chain_tip · new — singleton, coinbase maturity

Outputs stay flat — by design

The hot query "find spendable outputs for this user in the default basket" runs millions of times in production. v3 preserves the single-table scan. The outputs table keeps everything inline: txid, vout, userId, satoshis, basketId, lockingScript, spendable (cached), spentBy, isCoinbase, matures_at_height. No joins added.

SELECT * FROM outputs
WHERE user_id = ? AND spendable = true AND basketId = ?
ORDER BY satoshis DESC LIMIT 100;
-- Same plan as v2. Same indexes. Same latency.
02 — Processing FSM: 14 explicit states Tap any state to see its legal transitions. ›

v2 used a 6-state TransactionStatus enum that conflated user intent (NoSend), broadcast progress (Unmined, Callback), and finality (Completed). v3 separates concerns into a 14-state ProcessingStatus with an explicit transition table — illegal moves throw at the storage layer, not silently corrupt downstream queries.

Why this matters

  • seen vs. seen_multi. Provider acknowledgment is recorded at the granularity your monitoring needs. Multi-provider confirmation no longer collapses into the same bucket as a single 200 OK.
  • unconfirmed vs. confirmed. A merkle path that has not yet been validated against the chain tip is a different operational state than one that has. v3 keeps them apart so monitoring and replay logic don't have to.
  • frozen. Operator pause without losing position. Useful for compliance holds.
  • nonfinal. Live nLockTime transactions sit in their own state until eligibility, never poisoning the broadcast queue.
  • unfail. Admin retry override that wipes the rejection cause without losing audit history.
03 — Spendability: one rule, derived once The cached outputs.spendable flag is the only thing your wallet API has to trust. ›

v2 had spendability logic distributed across listOutputs, createAction, and the broadcast pipeline — every caller re-computed it. v3 derives it once, atomically, from the FSM state plus the chain tip:

// spendabilityRule.ts
const spendable =
  ProcessingSpendableStatus.includes(tx.processing) // sent, seen, seen_multi, unconfirmed, confirmed
  && output.spentBy === null
  && output.lockingScript !== null
  && (!output.isCoinbase || output.maturesAtHeight <= chainTip.height);

Every FSM transition triggers refreshOutputsSpendable for the affected txid. The flag in outputs.spendable is the canonical answer for every read path. No more drift between what the FSM says and what UTXO selection believes.

04 — Reorg-safe: confirmed stays confirmed The reorging state is a holding pen, not a verdict. ›

Spec §5: "A reorg SHALL never cause a transaction to be marked Invalid or DoubleSpend."

v2's status machine allowed completed → invalid when chaintracks signaled a different best chain — a silent retroactive flip of UTXO state. v3's FSM rejects that move. When a reorg is detected:

  1. confirmed → reorging (preserves current spendability).
  2. Monitor's reorg task re-queries chaintracks for the new merkle path.
  3. Outcome routes back to seen, unconfirmed, or confirmed. To land on invalid or doubleSpend, the row must first leave reorging — so the audit row attributes the rejection to a non-reorg cause.
User-visible effect
UTXO balances do not flap during normal reorg activity. Wallet UIs no longer need a "settling" debounce to mask v2's race conditions.
05 — Coinbase 100-block maturity Maturity is a column, not a callsite check. ›

v2 had no first-class coinbase handling — wallets that custodied mining rewards needed application-level guards to keep immature outputs out of UTXO selection. v3 ships:

  • outputs.isCoinbase — boolean flag set at internalization.
  • outputs.matures_at_height — block height at which the output becomes spendable (= mined-at-height + 100).
  • chain_tip singleton — current best-known height, written by Monitor.
  • Spendability rule applies the maturity guard automatically.

A backfillCoinbaseMaturity helper retro-fills the height column for any pre-existing coinbase outputs at cutover time. Mining-pool deployments get correct maturity without changing a single caller.

06 — Monitor: lease-based, multi-instance safe Run N Monitors. They will not step on each other. ›

v2's gotcha #5 from the README: "Don't run multiple monitor instances on the same storage simultaneously; they will race and cause state corruption." That constraint disappears in v3.

v3 introduces a monitor_lease table. Tasks like TaskCheckForProofs, TaskRebroadcast, and reorg detection each acquire a row-scoped lease before mutating transactions. Leases carry a TTL; abandoned leases are reclaimed on next pass.

// LeasedMonitorTask base class
async runOnce() {
  const claim = { taskName: 'proof-acquisition', ownerId: this.instanceId, ttlMs };
  const res = await txService.tryClaimLease(claim);
  if (!res.ok) return; // another instance holds it; back off
  try { await doWork(); }
  finally { await txService.releaseLease({ taskName: claim.taskName, ownerId: claim.ownerId }); }
}

Outcome: HA deployments without a coordinated leader; rolling restarts with zero risk of split-brain writes.

07 — tx_audit: every transition, on disk Replace "what happened to this txid?" forensics with a SQL query. ›

v2's proven_tx_reqs.history was a JSON blob — append-only inside the row, opaque to indexing, untestable. v3 replaces it with a real table:

CREATE TABLE tx_audit (
  auditId       INTEGER PRIMARY KEY,
  transactionId INTEGER NULL REFERENCES transactions(transactionId),
  actionId      INTEGER NULL REFERENCES actions(actionId),
  event         VARCHAR(64) NOT NULL,  -- 'processing.changed', 'proof.acquired', 'history.note', ...
  from_state    VARCHAR(16) NULL,       -- FSM source for transitions
  to_state      VARCHAR(16) NULL,       -- FSM target for transitions
  details_json  TEXT NULL,             -- JSON payload (provider, status, free-text 'what', etc.)
  created_at    DATETIME,
  updated_at    DATETIME
);
-- Indexes: event, transactionId, actionId

Useful queries become trivial: "show every state this txid was in", "which provider response is logged for this rejection" (details_json), "rate of * → doubleSpend in the last hour" (event = 'processing.changed' AND to_state = 'doubleSpend').

08 — Performance: the hot path is untouched Single-table outputs scan. Same indexes. Same plan. ›

The migration's hardest constraint was not regressing the spendable-outputs query. v3's outputs table preserves the exact same shape and indexes; the only added column is matures_at_height (NULL for non-coinbase).

The rollout gate (Week 2, staging soak) explicitly fails the cutover if p95 of findSpendableOutputs regresses more than 15% vs. the v2 baseline. In our staging soak it was unchanged within noise.

What does get slower (and where it doesn't matter)
listActions now joins actions ⨝ transactions instead of a single-table scan. That code path is run on user view-history loads — orders of magnitude colder than UTXO selection — and benefits from the FSM granularity it gains in return.

Migration mechanics

Six steps. One maintenance window.

The migration is additive-first: v3 code lands in production writing both schemas in parallel for a week before any cutover. The destructive rename runs once, in a scheduled window, and is fully scripted.

STEP 1
Deploy additive code
v3 release ships with the new tables (transactions_new, actions, tx_audit, monitor_lease, chain_tip). Reads still hit legacy.
STEP 2
Backfill
runBackfill populates the new tables from the legacy triad. Idempotent; can be paused and resumed.
STEP 3
Shadow-write
Every write hits both schemas. Divergence counter must hold at zero for 48 h before scheduling cutover.
STEP 4
Drain & pause Monitor
Maintenance window opens. Writers drained. Monitor stopped. monitor_lease verified empty.
STEP 5
runSchemaCutover
Renames legacy tables to *_legacy. Swaps transactions_new in. Remaps FKs in outputs, commissions, tx_labels_map. Idempotent.
STEP 6
Smoke tests, reopen
Eight smoke queries from the runbook. All must pass — orphan checks, row-count parity, and a synthetic create → transition → recordProof roundtrip.

Caller migration (what your app needs to do)

If you use SetupWallet or SetupClient, the answer is nothing. The public BRC-100 wallet interface is unchanged. createAction, listOutputs, internalizeAction, etc., all keep the exact same signatures and semantics.

If you reach into StorageProvider directly — for analytics, custom monitor tasks, or admin dashboards — these methods are net-new and you may want to adopt them:

v3 storage API additions// FSM-aware writes (all go through TransactionService)
await txService.create({ txid, processing: 'queued', rawTx, inputBeef });
await txService.transition({ transactionId, expectedFrom: 'queued', to: 'sending', provider: 'arc' });
await txService.recordProof({ transactionId, expectedFrom: 'unconfirmed', height, merklePath, merkleRoot, blockHash });
await txService.transitionMany({ transactionIds: ids, to: 'invalid', details: { reason: 'all providers rejected' } });

// per-user reads
await txService.findActionByReference(userId, reference);
await txService.findActionByUserTxid(userId, txid);
await txService.listActionsForUser({ userId, statusFilter, labelIds, labelQueryMode: 'all', limit, offset });
await txService.listOutputsForUser({ userId, basket, limit, offset });

// monitor leases (lease-based Monitor)
await txService.tryClaimLease({ taskName: 'proof-acquisition', ownerId, ttlMs });
await txService.renewLease({ taskName, ownerId, ttlMs });
await txService.releaseLease({ taskName, ownerId });

// audit + spendability
await refreshOutputsSpendable(knex, { transactionId });
await txService.recordHistoryNote(transactionId, { what: 'provider_response', provider: 'arc', status: '200' });

What breaks if you ignore the migration

Direct SQL against legacy table names
Any code still issuing SELECT … FROM proven_tx_reqs or reading transactions.status directly will throw the moment the cutover completes. The Week-3 production audit explicitly grep's the deploy for these references before opening the writer pool.

Rollout

Four-week, gated, reversible

Every week has explicit exit criteria. Any failed criterion blocks promotion to the next week. The rollback playbook is rehearsed once during the Week-3 soak.

WEEK 1Deploy & shadow

Additive code ships to staging then production. New tables created. Backfill rehearsed against a production snapshot. Performance baseline captured for the hot query.

WEEK 2Stage cutover + soak

Real runSchemaCutover executed on staging. Eight smoke tests must pass before reopening writers. Seven-day soak with k6 load test and full conformance suite on days 1, 3, 7.

WEEK 3Production cutover + monitor

Scheduled 02:00–04:00 UTC window. Writers drained, Monitor stopped, cutover run, smoke tests, writers reopened. Seven-day monitored soak with SRE on rollback rehearsal.

WEEK 4Decommission

Drop *_legacy tables. Remove shadow-write code. Archive backup to cold storage. Project retro.

Risk table (the ones that kept us up)

Cutover failure mid-rename Mitigation: runSchemaCutover is idempotent. Re-run resumes from transactions_legacy presence guard. ›

Every state during the cutover sequence is observable from SQL. If transactions_legacy exists but transactions_new does not, the rename completed; the FK remap may or may not have started. The runner detects this and resumes safely.

FK mismatch on outputs / tx_labels_map Mitigation: smoke tests 3.4 + 3.5 must return zero orphans before writers reopen. ›

The cutover remaps FK values by table-wide offset arithmetic. Two left-joins detect any row that survived with a dangling parent. Both queries are required to return 0; on failure the writer pool stays closed and the rollback path runs.

Performance regression on hot query Mitigation: k6 at 10× normal QPS on Week-2 staging; abort if p95 regresses >15%. ›

Single-table outputs scan is the load-bearing invariant. Performance Engineering signs off before the production cutover is scheduled; without the green light, the window does not open.

Rollback window blown Mitigation: *_legacy tables retained for 30 days; pre-cutover backup pinned in cold storage for the full Week-4 window. ›

Rollback is best-effort once writes have landed on the new transactions table. The runbook is explicit: if any post-cutover writes have occurred, the only safe path is forward-fix or restore-from-backup. The legacy tables are read-available for the full 30-day window so analytics queries can be cross-checked.

FAQ

Questions we hear most

Do I have to change application code? No — if you use Wallet, SetupWallet, or SetupClient as documented. ›

The BRC-100 wallet interface is unchanged: createAction, signAction, listActions, listOutputs, internalizeAction, relinquishOutput, getPublicKey, listCertificates, proveCertificate — same signatures, same semantics. The only thing you do is bump the npm version, redeploy, and run the migration.

Will the upgrade lose any data? No. The backfill carries every legacy row forward; orphan-checks run at cutover gate writers reopening. ›

Smoke tests 3.1 and 3.2 enforce row-count parity between transactions_legacy and the new transactions + actions pair. Smoke tests 3.4 and 3.5 enforce zero FK orphans. The cutover refuses to reopen the writer pool until all four return clean.

What about my IndexedDB clients? runIdbSchemaCutover runs on next app open; tracked by telemetry. ›

The browser/mobile bundles ship the same schema-cutover helper, scoped to a single IDB database. It runs the first time the upgraded client opens the wallet. A 14-day force-upgrade prompt is wired in for stragglers — disabled by default; opt in via settings.idbForceUpgradeAfterDays.

Can I run v2 and v3 against the same database? During the shadow window (Week 1), yes. After cutover, no. ›

Week 1 is explicitly designed for parallel running — additive code lands; both schemas update on every write; reads still hit legacy. Once cutover runs in Week 2/3, the legacy tables are renamed *_legacy and any v2 code path that reads them by their original names will throw.

What if I'm just integrating wallet-toolbox for the first time? Install v3 directly. There is no v2 obligation. ›

New deployments get the v3 schema from the first migrate() call. None of the legacy tables are created. None of the cutover machinery runs. You inherit the FSM, audit log, and lease-based Monitor on day one.

How big is the v3 npm release? No new runtime dependencies. Bundle size effectively flat across all three packages. ›

The lean-deps rule from CONTRIBUTING.md stands. v3 adds source files but no external packages. @bsv/wallet-toolbox, @bsv/wallet-toolbox-client, and @bsv/wallet-toolbox-mobile bundles are within a few KB of their v2.1.25 equivalents.

Where do I file issues during the rollout? github.com/bsv-blockchain/ts-stack — tag wallet-toolbox, prefix [v3]. ›

The rollout channel monitors the v3-prefixed issue queue daily. P1/P2 reports during a soak window route to SRE on-call automatically.
