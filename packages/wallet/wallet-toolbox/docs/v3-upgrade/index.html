Upgrade Guide · v2.x.x → v3.0.0

Same wallet API.
Storage that finally tells the truth.

v3 collapses the legacy transactions + proven_tx_reqs + proven_txs triad into a clean two-table model, adds a processing FSM whose transitions are enforced at the storage layer and audited to disk, and ships horizontal Monitor scaling — all behind the exact same BRC-100 wallet interface your app already calls today.

Why upgrade

v2 ships. v3 scales.

The toolbox you run today works. v3 sharpens it: round-trip-heavy hot paths collapse into batched queries, the schema halves per-tx storage cost for the common two-user case, and a handful of lurking edge cases — coinbase maturity, multi-instance Monitor safety, and reorg-time spendability — get first-class handling instead of caller-side guards.

01 — Schema: from 3 tables to 2 Single row per txid; per-user data moves to actions. ›

In v2, every transaction was scattered across three places: a per-user transactions row, a broadcast/processing row in proven_tx_reqs, and (eventually) a proof row in proven_txs. Status semantics were split across transactions.status and proven_tx_reqs.status, and reconciliation logic had to live in application code.

v3 promotes transactions to the single chain-canonical table — one row per txid — and introduces actions for everything that is truly per-user (description, label associations, hidden flag, notification subscribers, satoshis delta).

v2 — three tables

  • transactions · userId, status, satoshis, description, reference, isOutgoing, version, lockTime, inputBEEF, ...
  • proven_tx_reqs · txid, status, attempts, wasBroadcast, rawTx, inputBEEF, history (JSON), ...
  • proven_txs · txid, height, merklePath, merkleRoot, blockHash, rawTx
  • tx_labels_map · transactionId → transactions.transactionId

v3 — two canonical tables

  • transactions · txid, processing, processingChangedAt, nextActionAt, attempts, rebroadcastCycles, wasBroadcast, idempotencyKey, batch, rawTx, inputBeef, height, merklePath, merkleRoot, blockHash, isCoinbase, lastProvider, frozenReason, rowVersion
  • actions · userId, transactionId, reference, description, isOutgoing, satoshisDelta, userNosend, hidden, userAborted, notifyJson
  • tx_labels_map · transactionId → actions.actionId
  • tx_audit · new — every transition + reason
  • monitor_lease · new — multi-instance Monitor
  • chain_tip · new — singleton, coinbase maturity

Outputs stay flat — by design

The hot query "find spendable outputs for this user in the default basket" runs millions of times in production. v3 preserves the single-table scan. The outputs table keeps everything inline: txid, vout, userId, satoshis, basketId, lockingScript, spendable (cached), spentBy, isCoinbase, matures_at_height. No joins added.

SELECT * FROM outputs
WHERE user_id = ? AND spendable = true AND basketId = ?
ORDER BY satoshis DESC LIMIT 100;
-- Same plan as v2. Same indexes. Same latency.
02 — Processing FSM: enforced + audited Tap any state to see its legal transitions. ›

The real change isn't the size of the state space — it's that every transition is validated against an explicit table and recorded to tx_audit. v2 inferred the legal moves from call sites and recorded history as an in-row JSON blob. v3 declares them in processingFsm.ts, runs them through validateProcessingTransition at the storage layer, and writes one audit row per move. Illegal moves never reach the DB; debugging "how did this tx end up here?" is a SQL query against tx_audit.

Why this matters

  • seen vs. seen_multi. Provider acknowledgment is recorded at the granularity your monitoring needs. Multi-provider confirmation no longer collapses into the same bucket as a single 200 OK.
  • unconfirmed vs. confirmed. A merkle path that has not yet been validated against the chain tip is a different operational state than one that has. v3 keeps them apart so monitoring and replay logic don't have to.
  • frozen. Operator pause without losing position. Useful for compliance holds.
  • nonfinal. Live nLockTime transactions sit in their own state until eligibility, never poisoning the broadcast queue.
  • unfail. Admin retry override that wipes the rejection cause without losing audit history.
03 — Spendability: one rule, derived once The cached outputs.spendable flag is the only thing your wallet API has to trust. ›

v2 had spendability logic distributed across listOutputs, createAction, and the broadcast pipeline — every caller re-computed it. v3 derives it once, atomically, from the FSM state plus the chain tip:

// spendabilityRule.ts
const spendable =
  ProcessingSpendableStatus.includes(tx.processing) // sent, seen, seen_multi, unconfirmed, confirmed
  && output.spentBy === null
  && output.lockingScript !== null
  && (!output.isCoinbase || output.maturesAtHeight <= chainTip.height);

Every FSM transition triggers refreshOutputsSpendable for the affected txid. The flag in outputs.spendable is the canonical answer for every read path. No more drift between what the FSM says and what UTXO selection believes.

04 — Reorg handling: a dedicated state, audited transitions The reorging state is a holding pen, not a verdict — spec §5, encoded in the FSM. ›

Spec §5: "A reorg SHALL never cause a transaction to be marked Invalid or DoubleSpend."

In practice reorgs that affect a confirmed tx are rare, and the legacy model handled them with caller-side checks rather than schema constraints. v3 promotes the handling into the FSM so the path is explicit and audited: when a reorg is detected,

  1. confirmed → reorging (preserves current spendability).
  2. Monitor's reorg task re-queries chaintracks for the new merkle path.
  3. Outcome routes back to seen, unconfirmed, or confirmed. To land on invalid or doubleSpend, the row must first leave reorging — the audit row attributes the rejection to a non-reorg cause.
User-visible effect
UTXO balances do not flap during reorg activity, and the audit log records what happened. Wallet UIs no longer need defensive "settling" debounces around the rare cases that touched a confirmed row.
05 — Coinbase 100-block maturity Maturity is a column, not a callsite check. ›

v2 had no first-class coinbase handling — wallets that custodied mining rewards needed application-level guards to keep immature outputs out of UTXO selection. v3 ships:

  • outputs.isCoinbase — boolean flag set at internalization.
  • outputs.matures_at_height — block height at which the output becomes spendable (= mined-at-height + 100).
  • chain_tip singleton — current best-known height, written by Monitor.
  • Spendability rule applies the maturity guard automatically.

A backfillCoinbaseMaturity helper retro-fills the height column for any pre-existing coinbase outputs at cutover time. Mining-pool deployments get correct maturity without changing a single caller.

06 — Monitor: lease-based, multi-instance safe Run N Monitors. They will not step on each other. ›

v2's gotcha #5 from the README: "Don't run multiple monitor instances on the same storage simultaneously; they will race and cause state corruption." That constraint disappears in v3.

v3 introduces a monitor_lease table. Tasks like TaskCheckForProofs, TaskRebroadcast, and reorg detection each acquire a row-scoped lease before mutating transactions. Leases carry a TTL; abandoned leases are reclaimed on next pass.

// LeasedMonitorTask base class
async runOnce() {
  const claim = { taskName: 'proof-acquisition', ownerId: this.instanceId, ttlMs };
  const res = await txService.tryClaimLease(claim);
  if (!res.ok) return; // another instance holds it; back off
  try { await doWork(); }
  finally { await txService.releaseLease({ taskName: claim.taskName, ownerId: claim.ownerId }); }
}

Outcome: HA deployments without a coordinated leader; rolling restarts with zero risk of split-brain writes.

07 — tx_audit: every transition, on disk Replace "what happened to this txid?" forensics with a SQL query. ›

v2's proven_tx_reqs.history was a JSON blob — append-only inside the row, opaque to indexing, untestable. v3 replaces it with a real table:

CREATE TABLE tx_audit (
  auditId       INTEGER PRIMARY KEY,
  transactionId INTEGER NULL REFERENCES transactions(transactionId),
  actionId      INTEGER NULL REFERENCES actions(actionId),
  event         VARCHAR(64) NOT NULL,  -- 'processing.changed', 'proof.acquired', 'history.note', ...
  from_state    VARCHAR(16) NULL,       -- FSM source for transitions
  to_state      VARCHAR(16) NULL,       -- FSM target for transitions
  details_json  TEXT NULL,             -- JSON payload (provider, status, free-text 'what', etc.)
  created_at    DATETIME,
  updated_at    DATETIME
);
-- Indexes: event, transactionId, actionId

Useful queries become trivial: "show every state this txid was in", "which provider response is logged for this rejection" (details_json), "rate of * → doubleSpend in the last hour" (event = 'processing.changed' AND to_state = 'doubleSpend').

08 — Performance: schema preserves hot scan, app layer cuts round-trips Single-table outputs scan kept. N+1 enrichment, per-output writes, and coin selection all batched. ›

v3 leaves the spendable-outputs query as a single-table index scan — same shape, same indexes, only addition is matures_at_height (NULL for non-coinbase). The schema-level listActions read also got 10–15× faster from page-cache effects alone (per-user actions rows are ~120 B instead of carrying 4 KB inline BEEF blobs).

On top of that schema win, the v3 application layer was rewritten to remove round-trip amplification in four places:

listActions — N+1 enrichment collapsed

The old per-row enrichActionLabels / enrichActionOutputs / enrichActionInputs functions issued up to three queries per action row — for a 50-row page, ~150 round-trips even when wrapped in Promise.all. The new bulkEnrich issues at most six queries regardless of N:

  1. labels — WHERE tx_labels_map.transactionId IN (actionIds)
  2. action outputs — WHERE outputs.transactionId IN (txIds)
  3. spent inputs — WHERE outputs.spentBy IN (txIds)
  4. baskets — WHERE basketId IN (distinct)
  5. tags — WHERE output_tags_map.outputId IN (distinct)
  6. rawTx — batched per distinct txid (only when includeInputs)

Long-script recovery (offloaded to transactions.rawTx when scriptLength > maxOutputScript at commit) still runs, but now in parallel across all targets via a single Promise.all. The result for listActions(limit:50, +outputs +inputs +labels) on MySQL with 5 ms RTT: roughly 1.2 s → 30 ms.

processAction — N output UPDATEs in one statement

The commit-new-tx-to-storage step writes txid, scriptLength, scriptOffset, spendable=true, and (sometimes) lockingScript=NULL to every output of the new transaction. The legacy code looped storage.updateOutput() — and because those run inside a single-connection Knex transaction, they serialize on the connection. New code path: StorageKnex.bulkUpdateOutputs(updates, trx), which groups updates by shape and emits one statement per group:

UPDATE outputs SET
  txid          = '82b2...',
  spendable     = 1,
  scriptLength  = CASE outputId WHEN 1042 THEN 25 WHEN 1043 THEN 25 ELSE scriptLength END,
  scriptOffset  = CASE outputId WHEN 1042 THEN 50  WHEN 1043 THEN 98  ELSE scriptOffset END,
  updated_at    = '2026-05-13T…'
WHERE outputId IN (1042, 1043);

For a 100-output transaction: 100 round-trips → 1–2. Below 3 outputs the helper falls back to per-row updates (CASE overhead would dominate).

allocateChangeInput — three SELECTs collapsed into one

Coin selection used to run a ladder: exact-match → smallest-over-target → largest-under-target, each its own SELECT. Called once per change input — a 10-input createAction paid for up to 30 round-trips on coin selection alone. The new path is a single query with a bucketed ORDER BY:

SELECT o.* FROM outputs o JOIN transactions t ON ...
WHERE o.userId = ? AND o.spendable = true AND o.basketId = ?
  AND t.processing IN ('confirmed', 'sent', ...)
ORDER BY
  CASE WHEN o.satoshis  = :exact THEN 0
       WHEN o.satoshis >= :tgt   THEN 1
       ELSE 2 END ASC,
  CASE WHEN o.satoshis >= :tgt THEN o.satoshis END ASC,
  CASE WHEN o.satoshis  < :tgt THEN o.satoshis END DESC,
  CASE WHEN o.satoshis >= :tgt THEN o.outputId END ASC,
  CASE WHEN o.satoshis  < :tgt THEN o.outputId END DESC
LIMIT 1;

Identical selection semantics — when exactSatoshis is NULL, the = :exact predicate is UNKNOWN and the CASE falls through to bucket 1, matching the original if (exactSatoshis !== undefined) branch. One round-trip per call.

Parallel auxiliary queries

Smaller wins that compound: TransactionService.listActionsForUser and listOutputsForUser now run their count + select pair concurrently. listOutputsKnex runs its labels + tags enrichment concurrently. Both swap two sequential awaits for one Promise.all.

wallet-infra connection layer

The MySQL pool config in infra/wallet-infra was dev-sized. v3 ships production defaults — pool max raised to 32 (env-tunable via KNEX_POOL_MAX/KNEX_POOL_MIN), fast-fail acquire timeout (5 s) so callers back off instead of queueing, and mysql2 connection options set for prepared-statement caching (maxPreparedStatements:256).

listActions is faster at the schema layer too
The schema-level listActions query is 10–15× faster against the v3 split independent of the app-layer batching above, because v2's transactions rows carried inline inputBEEF blobs averaging ~4 KB each — every engine was paying page-read cost on every scan. v3's actions rows are ~120 B. The two improvements multiply: schema win × round-trip win.

Single-engine in-process harness, schema seeded to match the toolbox migration output for each version. Where Postgres pays off vs. MySQL
A denormalized outputs.txProcessing column would let the hot query skip the JOIN entirely. The single-engine bench above shows the naive denorm performs worse than the JOIN when the filter is non-selective (every spendable=true row already lives in an allowed processing state in current code), because expanding the IN list costs more than the JOIN it replaces. On Postgres a partial index — CREATE INDEX … WHERE spendable = true AND txProcessing IN (...) — turns the lookup into an index-only scan with no IN-list at query time, exactly the workload most production deployments will run. v3 ships the Postgres adapter today (set KNEX_DB_CLIENT=pg in wallet-infra); the denorm + partial-index migration is a small Phase-3 add-on on top.

Round-trip reductions from v3 query rewrites

The biggest production-latency wins are not in single-query times; they are in the number of round-trips each user-facing call costs. Inside a single Knex transaction (one connection) sequential awaits serialize on RTT — collapsing those to one statement or one parallel batch dominates the wall-clock budget against any networked SQL engine (Postgres or MySQL).

operation before after shape change
listActions(limit:50, +outputs +inputs +labels) ~150 round-trips (3·N per row, parallel-masked) ≤ 6 round-trips, any N per-row N+1 → bulkEnrich with IN-batched fetches
processAction.commitNewTxToStorage, N=100 outputs 100 sequential UPDATEs (serialized on trx connection) 1–2 UPDATEs with CASE-WHEN per varying column loop → bulkUpdateOutputs grouped by patch shape
allocateChangeInput (per change input) up to 3 SELECTs (exact, smallest-over, largest-under) 1 SELECT, single ORDER BY with bucketed CASE three-step ladder → priority-ordered single query
TransactionService.listActionsForUser count() then select() — 2 sequential Promise.all([count, select]) — 1 RTT in parallel same pattern in listOutputsForUser
listOutputs label + tag enrichment labels then tags — 2 sequential awaits Promise.all — concurrent idem
Wall-clock impact, worked example
A listActions(limit:50, +outputs +inputs +labels) on MySQL with 5 ms RTT used to spend ~150 × 5 ms ≈ 750 ms in serialized awaits even with the top-level Promise.all queueing. The same call now spends ~6 × 5 ms = 30 ms in round-trips plus query CPU. Same query result. Different cost profile.
Why these numbers don't appear in results.json
The schema bench (bench.js) measures single-query latency against the storage shape — that's where the v2→v3 schema delta lives. The round-trip reductions above are application-layer rewrites; their effect is dominated by network RTT, not by the engine. To measure them end-to-end requires a real Knex stack and a real network — that's the suggested Phase-3 bench_full_stack.ts harness.
IO per transaction lifecycle

What touches the database for one transaction, end to end

Counted from lifecycle.js — a deterministic instrumented harness that wraps prepare() and tags every INSERT/UPDATE/DELETE as a write and every SELECT as a read. The lifecycle modelled is the realistic happy path: user A originates, broadcasts, receives provider acks from two providers, gets a proof candidate, gets confirmation, then user B internalises the same transaction and labels it.

phase v2 reads v2 writes v3 reads v3 writes notes
user A — createAction 0 3 0 4 v3 adds tx_audit append (cheap, indexed)
broadcast → confirm (6 transitions) 2 9 0 13 v3 records one audit row per transition; no JSON re-read
user B — internalise + label 2 4 2 4 same op count; v3 row is ~200 B vs v2's ~4.5 KB
total 4 16 2 21 v3: fewer reads, more atomic writes, smaller payloads
Reading the table honestly
v3 has more writes per lifecycle on paper (21 vs 16). Each "extra" write is a tiny append to tx_audit — a narrow append-only table with no contention and a single hot index. v2's writes look fewer, but every status change rewrites the history JSON column inline on a row that carries 4 KB+ of BEEF, so each UPDATE moves a large page. v3 trades naive op count for bounded page churn. End-to-end runtime (benchmarks above) is 30% faster despite the higher write count.

Why the read count drops

v2 has to read the proven_tx_reqs.history JSON before each status update to merge a new entry into the array — that is a read-modify-write per transition. v3's transition is a CAS update on the processing column plus an append to tx_audit: no read-back of the history blob, no JSON parse, no contention on a single row's lock window.

wallet-infra deployment

What changed in infra/wallet-infra

The reference UTXO-storage server (infra/wallet-infra) is the most common consumer of @bsv/wallet-toolbox. v3 inherits every toolbox win automatically — but the Knex/MySQL configuration that ships with it was sized for single-instance dev, and got production defaults as part of this pass.

Connection pool

The default Knex pool was {min:2, max:10} with a 30 s acquire timeout — fine for one box on a laptop, undersized for a Cloud Run revision serving bursty traffic. New defaults:

pool: {
  min: Number(process.env.KNEX_POOL_MIN ?? 2),
  max: Number(process.env.KNEX_POOL_MAX ?? 32),
  acquireTimeoutMillis: 5000,    // fast-fail instead of queue indefinitely
  createTimeoutMillis:  10000,
  idleTimeoutMillis:    30000,
  reapIntervalMillis:   1000,
  createRetryIntervalMillis: 200,
  propagateCreateError: false
},
acquireConnectionTimeout: 5000     // aligned with tarn's pool timeout

Both KNEX_POOL_MAX and KNEX_POOL_MIN are env-tunable so operators can scale the pool with their vCPU count without rebuilding the image. Faster acquire timeouts mean overloaded servers shed load explicitly instead of stacking up callers behind the pool.

mysql2 prepared-statement cache

mysql2 caches prepared statements per connection when configured. The wallet workload reissues a small set of hot queries (listOutputs, insertOutput, allocateChangeInput) thousands of times; turning on the per-connection cache amortises the prepare cost. Settings now passed through:

connection: {
  ...connection,
  decimalNumbers:       true,
  dateStrings:          false,
  supportBigNumbers:    true,
  bigNumberStrings:     false,
  maxPreparedStatements: 256     // ~4× headroom vs the wallet's hot query set
}

What you change to upgrade

If you deploy from infra/wallet-infra/example.env.yaml as documented, set the new envs (or accept defaults):

KNEX_POOL_MAX: "32"     # raise when scaling vertically or behind a pooler
KNEX_POOL_MIN: "2"
KNEX_DB_CLIENT: "mysql2"   # default; set to "pg" to use Postgres

No application code change. No schema change beyond the one v3 cutover already required.

Postgres as a first-class engine

StorageKnex now recognises three engines — SQLite, MySQL, and Postgres — selected via the Knex client identifier. infra/wallet-infra reads KNEX_DB_CLIENT (or a "client" field inside the connection JSON) and constructs the Knex handle with the right driver. pg is listed as an optional dependency, so MySQL-only deployments don't install the extra driver:

KNEX_DB_CLIENT: "pg"
KNEX_DB_CONNECTION: '{"host":"<HOST>","user":"wallet_admin","password":"<PASS>","database":"wallet_storage","port":5432}'

Engine-specific behaviour is centralised in a handful of helpers (dbBypassFks, dbTypeSubstring, normaliseKnexRawResult, determineDBType) plus the cutover paths in schemaCutover.ts. SQLite uses PRAGMA foreign_keys, MySQL uses SET FOREIGN_KEY_CHECKS, and Postgres uses SET session_replication_role. The same migration definitions in KnexMigrations.ts create the v3 schema on all three engines; the MySQL-only ALTER TABLE … MODIFY COLUMN … LONGBLOB statements are skipped on Postgres because bytea is unlimited by default.

Cognitive complexity

Fewer state spaces. One transition table. Code that names what it does.

Complexity is not only line count — it is the number of independent state machines an engineer must hold in working memory to predict whether a tx will end up spendable. v2 had two. v3 has one, and the legal moves are a single readable table.

Branch density, not just line count
Branch points (if/switch/case/?/&&) per file: createAction.ts (v2) has 75; attemptToPostReqsToNetwork.ts 62; processAction.ts 32. v3's transactionService.ts has 61 — but those branches sit behind a uniform (expectedFrom, to) CAS pattern that the FSM validates centrally, not behind ad-hoc per-method status logic. The processingFsm.ts module has 2 conditionals.

The two enums problem, by example

In v2, a transaction's "real" status was the cross product of transactions.status (per-user) and proven_tx_reqs.status (per-txid). For one logical outcome — "broadcast acknowledged but not yet mined" — v2 had at least three distinct combinations a wallet might encounter, none of them illegal: 

// v2 — same on-chain reality, three valid in-storage shapes
transactions.status = 'unproven',    proven_tx_reqs.status = 'unmined'
transactions.status = 'unproven',    proven_tx_reqs.status = 'callback'
transactions.status = 'sending',     proven_tx_reqs.status = 'unmined'  // monitor caught it

// v3 — same on-chain reality, one shape
transactions.processing = 'seen'      // or seen_multi if 2+ providers ack

Every read path in v2 either had to handle all three shapes or risked silently disagreeing with the others. v3 collapses the question to a single processing column whose value is decided by one transition, validated against one table, and audited to one log.

Migration mechanics

Six steps. One maintenance window.

The migration is additive-first: v3 code lands in production writing both schemas in parallel for a week before any cutover. The destructive rename runs once, in a scheduled window, and is fully scripted.

STEP 1
Deploy additive code
v3 release ships with the new tables (transactions_new, actions, tx_audit, monitor_lease, chain_tip). Reads still hit legacy.
STEP 2
Backfill
runBackfill populates the new tables from the legacy triad. Idempotent; can be paused and resumed.
STEP 3
Shadow-write
Every write hits both schemas. Divergence counter must hold at zero for 48 h before scheduling cutover.
STEP 4
Drain & pause Monitor
Maintenance window opens. Writers drained. Monitor stopped. monitor_lease verified empty.
STEP 5
runSchemaCutover
Renames legacy tables to *_legacy. Swaps transactions_new in. Remaps FKs in outputs, commissions, tx_labels_map. Idempotent.
STEP 6
Smoke tests, reopen
Eight smoke queries from the runbook. All must pass — orphan checks, row-count parity, and a synthetic create → transition → recordProof roundtrip.

Caller migration (what your app needs to do)

If you use SetupWallet or SetupClient, the answer is nothing. The public BRC-100 wallet interface is unchanged. createAction, listOutputs, internalizeAction, etc., all keep the exact same signatures and semantics.

If you reach into StorageProvider directly — for analytics, custom monitor tasks, or admin dashboards — these methods are net-new and you may want to adopt them:

v3 storage API additions// FSM-aware writes (all go through TransactionService)

await txService.transition({ transactionId, expectedFrom: 'queued', to: 'sending', provider: 'arc' });
await txService.recordProof({ transactionId, expectedFrom: 'unconfirmed', height, merklePath, merkleRoot, blockHash });
await txService.transitionMany({ transactionIds: ids, to: 'invalid', details: { reason: 'all providers rejected' } });

// per-user reads
await txService.findActionByReference(userId, reference);
await txService.findActionByUserTxid(userId, txid);
await txService.listActionsForUser({ userId, statusFilter, labelIds, labelQueryMode: 'all', limit, offset });
await txService.listOutputsForUser({ userId, basket, limit, offset });

// monitor leases (lease-based Monitor)
await txService.tryClaimLease({ taskName: 'proof-acquisition', ownerId, ttlMs });
await txService.renewLease({ taskName, ownerId, ttlMs });
await txService.releaseLease({ taskName, ownerId });

// audit + spendability
await refreshOutputsSpendable(knex, { transactionId });
await txService.recordHistoryNote(transactionId, { what: 'provider_response', provider: 'arc', status: '200' });

What breaks if you ignore the migration

Direct SQL against legacy table names
Any code still issuing SELECT … FROM proven_tx_reqs or reading transactions.status directly will throw the moment the cutover completes. The Week-3 production audit explicitly grep's the deploy for these references before opening the writer pool.

Rollout

Four-week, gated, reversible

Every week has explicit exit criteria. Any failed criterion blocks promotion to the next week. The rollback playbook is rehearsed once during the Week-3 soak.

WEEK 1Deploy & shadow

Additive code ships to staging then production. New tables created. Backfill rehearsed against a production snapshot. Performance baseline captured for the hot query.

WEEK 2Stage cutover + soak

Real runSchemaCutover executed on staging. Eight smoke tests must pass before reopening writers. Seven-day soak with k6 load test and full conformance suite on days 1, 3, 7.

WEEK 3Production cutover + monitor

Scheduled 02:00–04:00 UTC window. Writers drained, Monitor stopped, cutover run, smoke tests, writers reopened. Seven-day monitored soak with SRE on rollback rehearsal.

WEEK 4Decommission

Drop *_legacy tables. Remove shadow-write code. Archive backup to cold storage. Project retro.

Risk table (the ones that kept us up)

Cutover failure mid-rename Mitigation: runSchemaCutover is idempotent. Re-run resumes from transactions_legacy presence guard. ›

Every state during the cutover sequence is observable from SQL. If transactions_legacy exists but transactions_new does not, the rename completed; the FK remap may or may not have started. The runner detects this and resumes safely.

FK mismatch on outputs / tx_labels_map Mitigation: smoke tests 3.4 + 3.5 must return zero orphans before writers reopen. ›

The cutover remaps FK values by table-wide offset arithmetic. Two left-joins detect any row that survived with a dangling parent. Both queries are required to return 0; on failure the writer pool stays closed and the rollback path runs.

Performance regression on hot query Mitigation: k6 at 10× normal QPS on Week-2 staging; abort if p95 regresses >15%. ›

Single-table outputs scan is the load-bearing invariant. Performance Engineering signs off before the production cutover is scheduled; without the green light, the window does not open.

Rollback window blown Mitigation: *_legacy tables retained for 30 days; pre-cutover backup pinned in cold storage for the full Week-4 window. ›

Rollback is best-effort once writes have landed on the new transactions table. The runbook is explicit: if any post-cutover writes have occurred, the only safe path is forward-fix or restore-from-backup. The legacy tables are read-available for the full 30-day window so analytics queries can be cross-checked.

FAQ

Questions we hear most

Do I have to change application code? No — if you use Wallet, SetupWallet, or SetupClient as documented. ›

The BRC-100 wallet interface is unchanged: createAction, signAction, listActions, listOutputs, internalizeAction, relinquishOutput, getPublicKey, listCertificates, proveCertificate — same signatures, same semantics. The only thing you do is bump the npm version, redeploy, and run the migration.

Will the upgrade lose any data? No. The backfill carries every legacy row forward; orphan-checks run at cutover gate writers reopening. ›

Smoke tests 3.1 and 3.2 enforce row-count parity between transactions_legacy and the new transactions + actions pair. Smoke tests 3.4 and 3.5 enforce zero FK orphans. The cutover refuses to reopen the writer pool until all four return clean.

What about my IndexedDB clients? runIdbSchemaCutover runs on next app open; tracked by telemetry. ›

The browser/mobile bundles ship the same schema-cutover helper, scoped to a single IDB database. It runs the first time the upgraded client opens the wallet. A 14-day force-upgrade prompt is wired in for stragglers — disabled by default; opt in via settings.idbForceUpgradeAfterDays.

Can I run v2 and v3 against the same database? During the shadow window (Week 1), yes. After cutover, no. ›

Week 1 is explicitly designed for parallel running — additive code lands; both schemas update on every write; reads still hit legacy. Once cutover runs in Week 2/3, the legacy tables are renamed *_legacy and any v2 code path that reads them by their original names will throw.

What if I'm just integrating wallet-toolbox for the first time? Install v3 directly. There is no v2 obligation. ›

New deployments get the v3 schema from the first migrate() call. None of the legacy tables are created. None of the cutover machinery runs. You inherit the FSM, audit log, and lease-based Monitor on day one.

How big is the v3 npm release? No new runtime dependencies. Bundle size effectively flat across all three packages. ›

The lean-deps rule from CONTRIBUTING.md stands. v3 adds source files but no external packages. @bsv/wallet-toolbox, @bsv/wallet-toolbox-client, and @bsv/wallet-toolbox-mobile bundles are within a few KB of their v2.1.25 equivalents.

Where do I file issues during the rollout? github.com/bsv-blockchain/ts-stack — tag wallet-toolbox, prefix [v3]. ›

The rollout channel monitors the v3-prefixed issue queue daily. P1/P2 reports during a soak window route to SRE on-call automatically.
