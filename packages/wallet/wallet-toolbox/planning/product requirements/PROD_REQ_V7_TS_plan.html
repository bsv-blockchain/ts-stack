TypeScript V7 wallet schema migration plan

A self-contained developer plan for moving the current TypeScript wallet-toolbox schema into V7: canonical transaction state, per-user action state, and a denormalized outputs table that keeps spendable-output selection fast.

proven_tx_reqs + proven_txs Processing queue and final Merkle proof sources.
->
transactions One canonical network lifecycle row per txid.
->
actions + outputs User-facing action state plus single-table UTXO lookup.
1 canonical transaction record per txid.
2 proof and queue tables merged into transactions.
0 joins on the common spendable outputs query.
30d recommended legacy-table retention window.

Current TypeScript Schema

The migration starts from the current TS wallet-toolbox schema only. Earlier draft schemas are intentionally out of scope.

Per-user

transactions

Current user-facing transaction view with status, description, labels, satoshis, references, and other wallet metadata.

Queue

proven_tx_reqs

Per-txid broadcast and processing queue with status, attempts, wasBroadcast, rebroadcastAttempts, inputBEEF, rawTx, and history JSON.

Proof

proven_txs

Final proof storage for height, index, merklePath, blockHash, merkleRoot, and rawTx.

Outputs

outputs

Stores output data. Some deployments also use user_utxos as a side table for fast per-user queries.

Tags

output_tags + maps

Output tags and mappings remain unchanged in V7.

Labels

tx_labels + maps

Transaction labels stay per-user, but tx_labels_map will point at actionId after migration.

Spendable-output lookup remains join-free

The common default-basket query stays a single-table index scan.

SELECT * FROM outputs
WHERE user_id = ?
  AND spendable = true
  AND basketId = default_basket
ORDER BY satoshis DESC
LIMIT 100;

Supporting tables

chain_tip stores current height. tx_audit records event history with transactionId and actionId scope. monitor_lease coordinates monitoring workers.

Migration Guide

Create new tables first, backfill them from the current TS sources, then perform a short atomic rename cutover.

01

Create V7 tables

Create the replacement tables without mutating existing production tables.

CREATE TABLE actions ( ... full definition as in target actions ... );
CREATE TABLE transactions_new ( ... full definition as in target transactions ... );
CREATE TABLE outputs_new ( ... full definition as in target outputs ... );
02

Backfill actions from current transactions

Move the current per-user transaction data into actions and convert nosend into user_nosend.

INSERT INTO actions (
  userId, transactionId, reference, description, is_outgoing,
  satoshis_delta, user_nosend, hidden, user_aborted,
  notify_json, created_at, updated_at
)
SELECT
  userId, transactionId, reference, description, isOutgoing,
  satoshis,
  CASE WHEN status = 'nosend' THEN TRUE ELSE FALSE END,
  FALSE, FALSE, notify_json, created_at, updated_at
FROM transactions;
03

Merge proven_tx_reqs and proven_txs

For every txid in either source table, create one transactions_new row. Processing comes from proven_tx_reqs.status; proof fields come from proven_txs.

completed -> proven unconfirmed -> unconfirmed unmined -> sent sending -> sending invalid -> invalid doubleSpend -> doubleSpend
04

Migrate outputs

Materialize user-specific output rows by joining outputs to current transactions.

INSERT INTO outputs_new (
  txid, vout, user_id, amount, locking_script, spendable,
  basketId, label, description, is_coinbase, matures_at_height,
  spent_by, created_at, updated_at
)
SELECT
  o.txid, o.vout, t.userId, o.satoshis, o.lockingScript,
  o.spendable, o.basketId, o.label, o.description,
  o.is_coinbase, o.matures_at_height, o.spentBy,
  o.created_at, o.updated_at
FROM outputs o
JOIN transactions t ON t.transactionId = o.transactionId;
05

Re-point references

Update tx_labels_map to target actionId. Move any references from proven_tx_reqs or proven_txs to the new transactions table.

06

Atomic cutover

Promote V7 tables and move old TS tables aside inside a transaction or brief maintenance window.

BEGIN;
  RENAME TABLE transactions TO transactions_old;
  RENAME TABLE transactions_new TO transactions;
  RENAME TABLE outputs TO outputs_old;
  RENAME TABLE outputs_new TO outputs;
  RENAME TABLE proven_tx_reqs TO proven_tx_reqs_old;
  RENAME TABLE proven_txs TO proven_txs_old;
COMMIT;
07

Cleanup after verification

Drop proven_tx_reqs_old, proven_txs_old, transactions_old, and outputs_old only after the conformance suite and performance checks pass. Rebuild required indexes.

Spendability Derivation

The cached outputs.spendable boolean remains, but it is now derived against the canonical transaction processing state.

Cached expression

Refresh this atomically on processing, spend, script, maturity, or chain-tip changes.

spendable = (
  transactions.processing IN (
    'sent', 'seen', 'seen_multi', 'unconfirmed', 'proven'
  )
)
AND outputs.spent_by IS NULL
AND outputs.locking_script IS NOT NULL
AND (
  NOT outputs.is_coinbase
  OR outputs.matures_at_height <= chain_tip.height
)
queued sending sent seen seen_multi unconfirmed proven reorging invalid doubleSpend unfail frozen nosend nonfinal

Merge correctness

Invariant

Output speed

Spendable selection remains a single-table index scan against outputs.

Invariant

Rollout Recommendation

Use a conservative migration path: staging data first, measured hot-path performance, V7 writes, then delayed cleanup.
